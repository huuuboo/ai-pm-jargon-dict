Why · 为什么需要这本词典

不懂术语，
就听不懂会议。

AI 行业每周都在制造新词。SFT、RLHF、RAG、MCP、Agentic Workflow——你听不懂，不是因为你笨，是因为这些词大多在英文论文里诞生，被翻译成中文之前已经被中国 PM 们直接用着开会了。

这本词典不教你写代码、不教你调模型。它只做一件事：把每个高频术语翻译成你能立刻用到的判断——什么时候该上 RAG、什么时候该走 Fine-tuning、什么时候该说"我们没有护城河"。

Foundation
Model
基座模型
essence · 一句话本质
读完了全人类书的"毛坯学霸"——什么都懂，还没被分配具体工作。
concept · 概念

用海量数据（万亿级 Token）预训练出来的通用大模型，本身什么具体任务都不会做，但什么都能学。GPT-4、Claude、Llama、Qwen 都是基座模型。

讨论技术选型、自研 vs 调用第三方、模型能力边界时会用。

scripts · 话术示例

Term · No.02
SFT
监督微调 · Supervised Fine-Tuning
essence · 一句话本质
教模型把它已经知道的事，用人类喜欢的方式说出来。
concept · 概念

预训练完的模型像个"什么都知道但不会好好说话"的人，SFT 用人工标注的高质量问答对教它"怎么正确回答"。比如几万条"用户问→标准答案"示例，让它学会对话的格式和规范。

context · 使用语境

讨论模型效果优化、标注团队投入、私有化部署时会用。

scripts · 话术示例
"通用模型在我们金融场景下表现不行，得准备一批行业 QA 数据做 SFT。"
"SFT 数据不在多在精——一千条高质量比一万条垃圾效果好。"

DPO
直接偏好优化 · Direct Preference Optimization
essence · 一句话本质
RLHF 的简化版——不训奖励模型，直接用偏好数据调模型本身。

Term · No.06
Scaling
Law
缩放定律
essence · 一句话本质
参数 × 数据 × 算力一起放大，模型能力可预测地涨——这是大厂烧钱的理论基础。
concept · 概念

OpenAI 在 2020 年提出的经验规律：模型参数量、训练数据量、算力三者按比例扩大，效果就会持续提升，而且可预测。这条定律是过去几年大模型军备竞赛的理论基础——"只要砸够钱，模型就会变强"。

讨论 AI 行业趋势、投资逻辑、模型发展路径时会用。

scripts · 话术示例
"Scaling Law 还没撞墙，所以大厂还在拼命堆算力。"
"如果 Scaling Law 失效了，那整个行业的投资逻辑要重写。"

Reasoning
推理能力
essence · 一句话本质
模型像人一样多步思考再回答，准确率高但成本是普通的 5-10 倍。

模型像人一样进行逻辑推理、多步思考的能力，而不是简单模式匹配。OpenAI 的 o1、DeepSeek-R1 是专门强化了推理能力的模型，会先"想一想"再回答，准确率大幅提升但速度更慢、成本更高。

scripts · 话术示例

Term · No.08
CoT
思维链 · Chain of Thought
essence · 一句话本质
强迫模型先「写解题过程」再给答案，准确率立刻飙升。
concept · 概念

让模型把推理过程一步步写出来而不是直接蹦答案。就像考试时要求"写出解题过程"——写过程的准确率比直接写答案高很多。可以通过 Prompt 引导（"请一步步思考"），也可以训练内化到模型里。

context · 使用语境

讨论 Prompt 设计、模型准确率优化时会用。

scripts · 话术示例
"这个数学计算任务直接问准确率才 60%，加 CoT 提示就到 90%。"
"在 System Prompt 里加一句'请分步推理后再给出最终答案'，效果立竿见影。"

Term · No.09
Token
词元 / 计费单位
essence · 一句话本质
AI 的「字数」和「计费单位」——理解 Token 就理解 AI 产品的成本结构。
concept · 概念

模型处理文本的最小单位。英文约 1 词 ≈ 1-1.5 Token，中文约 1 字 ≈ 1.5-2 Token。所有大模型 API 都按 Token 计费，输入和输出分别计价（输出通常贵 2-4 倍）。

context · 使用语境

讨论成本核算、定价策略、产品设计（如字数限制）时无处不在。

scripts · 话术示例
"这个功能每次调用平均消耗 3000 Token，按 GPT-4o 算，一次成本大概 3 分钱。"
"用户发了一份 10 万字的文档，光输入就要 15 万 Token，成本扛不住。"

Term · No.10
Context
Window
上下文窗口
essence · 一句话本质
模型一次能"看到"的内容上限——它的工作记忆容量。
concept · 概念

模型一次能处理的 Token 数量上限。窗口内的内容它能理解和引用，窗口外的就忘了。GPT-4 Turbo 是 128K，Claude 是 200K，Gemini 到了 1M。但"能放进去"≠"能用好"——长上下文中间部分容易丢失（Lost in the Middle）。

Temperature
温度
essence · 一句话本质
控制模型「放飞程度」的旋钮——0 严谨可复现，1 有创意。
concept · 概念

控制模型输出随机性的参数，范围通常 0-2。Temperature=0 时给最确定的答案（适合数据提取、分类），Temperature=1 时比较有创意（适合写文案、头脑风暴）。产品经理必须知道不同场景该怎么调。

模型层 · 5 个产品决策

读完上面 14 个词，你应该能在会议上回答以下问题。

Decision 01
自研基座 vs 调用 API？
99% 创业公司不要自训基座（几个亿起步）。在 Qwen / Llama 这些开源基座上微调，或直接调闭源 API 验证 PMF——切开源是为降本，不是为情怀。
Decision 02
需不需要做微调？
先试 Prompt + Few-shot，搞不定再上 RAG，再搞不定才微调。微调最贵的是数据准备不是训练本身——一千条高质量胜过一万条垃圾。
Decision 03
推理模型 vs 普通模型？
看任务复杂度。法律审查、代码 debug、多步分析 → 推理模型。简单 QA、改写、分类 → 普通模型。Token 成本差 5-10 倍，算清 ROI 再上。
Decision 04
Temperature 该调多少？
客服 / 数据提取 / 分类 → 0-0.3；通用对话 → 0.5-0.7；创意写作 / 头脑风暴 → 0.8-1.0。别和 Top-P 同时调太猛，会出胡言乱语。
Decision 05
如何治理幻觉？
高风险场景（医疗、法律、金融）必须 RAG + Citation + 置信度展示三件套。中低风险用 System Prompt 兜底（"不确定就说不知道"）。永远不要只靠模型自己。
Decision 06
长文档怎么处理？
≤ 模型上下文窗口 → 直接塞；超出 → RAG。注意 Lost in the Middle——超长上下文中间部分会丢，长合同审查仍要分段。

Decision Checklist
Chapter Recap · 章末决策清单

数据层 · 5 个产品决策

读完上面 11 个词，这些数据相关的决策你应该不再含糊。

Decision 01
RAG vs Fine-tuning vs 长上下文？
企业知识 + 频繁更新 → RAG；企业风格/语调要内化 → 微调；文档 ≤ 模型上下文 → 直接塞。三者不是互斥，常组合。
Decision 02
RAG 效果差，先查什么？
先查 Chunking（怎么切的），再查 Embedding 模型质量，最后才换大模型。模型很少是 RAG 失败的根因。
Decision 03
数据飞轮真的转起来了吗？
问三个问题：用户反馈数据存哪？谁在标？标完进了哪个训练集？任何一环断了都不是真飞轮。
Decision 04
合成数据可不可以用？
早期 / 数据稀缺时可以，但要混真实数据稀释（10:1 比例）防 Model Collapse。永远不要 100% 合成。
Decision 05
每次模型迭代怎么知道好坏？
Golden Set 是基础设施。没有的项目立刻建——300-500 条覆盖核心场景的"标准答案"，每次发版必跑。
Decision 06
数据护城河怎么建？
三个来源：① 用户行为数据（产品规模）；② 行业专属数据（领域 know-how）；③ 持续标注（运营投入）。三者中至少占一项才说得上有 Moat。

Decision 01
Chatbot vs Workflow vs Agent？
输入输出明确 → Chatbot；流程稳定可枚举 → Workflow；目标导向、路径不确定 → Agent。Agent 是潮流但成本和不确定性最高，别盲目上。
Decision 02
效果差，先动 Prompt 还是换模型？
优先级：Prompt → Few-shot → RAG → 微调 → 换模型。先动便宜的，最后才动贵的。换模型是最后选项，不是第一反应。
Decision 03
用户信任怎么建？
三件套：Citation（带出处）+ Confidence（标置信度）+ 渐进式信任（先低风险后高风险）。To B 高风险场景三个都要，To C 至少要 Citation。
Decision 04
该不该上 Streaming？
几乎所有面向人的对话场景都该上。例外：要展示结构化结果（表格/JSON）、要做后处理过滤的场景，可以等全部生成完再渲染。
Decision 05
长期记忆要不要做？
看你的产品是工具型还是陪伴型。陪伴型 / 个性化推荐 / 助理类 → 必做；一次性查询工具 → 可不做。注意隐私和"记错的尴尬"。
Decision 06
Eval 集怎么搭？
从真实用户日志里挑 100-300 条覆盖核心场景 + 边缘 case，人工标"理想答案"。每次发版必跑，回退指标超阈值就阻断上线。

用户层的核心问题不是「拉多少」，是「留多少 / 用多深」。

Decision 01
核心指标看什么？
不要只看 DAU。AI 产品至少看：任务完成率 + 留存（D1/D7/D30）+ Engagement Depth（人均轮次/任务数）。三个一起看才是真相。
Decision 02
Aha Moment 怎么定义？
看高留存用户的早期行为差异——哪个动作完成的用户次日留存高 30%+，那个就是 Aha。然后逆推 Onboarding 让新用户尽快做到。
Decision 03
留存断崖怎么找？
画 D1/D3/D7/D14/D30 留存曲线，找到下跌最陡的那段。再做用户访谈/行为分析定位原因——通常是「新鲜感消退 + 没新场景」。
Decision 04
为谁优化？大众 vs Power User？
早期为 Power User 优化（找出深度场景），中期同时服务两边（Onboarding 给新人，高级功能给老人），后期警惕 Power User 陷阱。
Decision 05
冷启动怎么破局？
100 种子用户深度陪跑 > 10 万路人用户。前期手动喂数据、人肉跑 RAG、产品经理亲自客服——做不可规模化的事，把数据飞轮转起来。
Decision 06
怎么判断有没有 PMF？
三件套：① 留存平稳（不是不停下跌）；② 自发推荐（NPS / 自然增长）；③ 用户骂"缺功能"而不是骂"难用"。三个都中才算找到。

Decision Checklist
Chapter Recap · 章末决策清单

商业层 · 6 个产品决策

商业层是离亏损最近的一层。下面每条都是花钱前必须回答的问题。

Decision 01
该不该做这个 AI 产品？
三件套必算：① Token 单次成本；② 预估 ARPU；③ Unit Economics 算正？三个数算下来 ARPU > 2 × 单位成本才能动手。
Decision 02
定多少钱？怎么定？
先看 Token 成本兜底，再看竞品定价，最后看 WTP（用户调研）。三个值取交集。To C 关心绝对价（19/29/39），To B 关心 ROI（替代多少人力）。
Decision 03
护城河在哪？
问自己一个问题：如果模型方明天做一个一模一样的功能，你还剩什么？答不出来就是没护城河——快速找数据 / 场景 / 用户习惯三件套之一。
Decision 04
要不要烧钱拉新？
先确认 PMF + LTV:CAC > 3。两个都不满足就别拉——花的不是用户增长，是钱给 GPU 厂商充值。
Decision 05
通用 vs 垂直 选哪个？
非 BAT 体量公司一律选垂直。垂直壁垒来自行业数据 + 客户关系，是 OpenAI 都拿不走的——这是创业公司唯一能跑赢巨头的赛道。
Decision 06
毛利怎么打到健康？
三招组合：① 模型路由（小问题用小模型）；② 缓存（重复请求不重算）；③ 蒸馏（把大模型能力蒸到小模型上）。能把毛利从 30% 拉到 65%。

工程架构 · 5 个产品决策

PM 不写代码，但要懂工程同事在吵什么——下面是 PM 应该参与的工程决策。

Decision 01
体感慢，先优化什么？
先看 TTFT（首 Token 时间），不看总耗时。TTFT < 1.5s 用户体感"快"。配合 Streaming，延迟感能降一半。
Decision 02
成本高怎么降？
三招：① 模型路由（70% 简单问题给小模型）；② 缓存（重复请求不重算）；③ Prompt 精简（输出 Token 是输入的 2-4 倍贵）。
Decision 03
端侧 vs 云端 怎么选？
隐私敏感（医疗 / 财务 / 私人对话）+ 简单任务 → 端侧；复杂推理 + 大上下文 → 云端；中等场景 → 端云协同。
Decision 04
线上 Prompt 谁管？
必须有 Prompt Management 系统：版本号 + 责任人 + A/B 实验 + 一键回滚。手动改 Prompt 上线 = 灾难前夜。
Decision 05
线上质量怎么保证？
Observability 三件套：延迟/错误率/成本（系统指标）+ 幻觉率/拒答率/用户满意度（业务指标）+ 自动告警。出事前发现，不是出事后救火。
Decision 06
什么时候上 LLMOps？
团队 ≥ 5 人 / Prompt ≥ 10 个 / 月迭代 ≥ 4 次——三个里任意中一个就该上。再晚就是技术债。

免费版基础模型，付费版高级模型 + 联网。让用户体验到价值差再升级。
Rate Limit
限速 · Ch.V
限制单位时间调用次数。控成本 + 制造升级动力。
CAC Payback
获客成本回收周期 · Ch.V
花的获客成本要几个月赚回。AI 因教育 + 试用补贴，回收周期偏长。
NRR
净收入留存 · Ch.V
超 100% 老客户在加钱，< 100% 在流失。顶级 SaaS 120%+。
WTP
付费意愿 · Ch.V
用户愿意为某个功能付多少钱。定价前必须通过调研 / A/B 摸清。
Price Sensitivity
价格敏感度 · Ch.V
To C 对几块差价敏感（29 vs 39 差很多）；To B 看 ROI 不看单价。
订阅疲劳
Subscription Fatigue · Ch.V
用户已订太多 SaaS，再加一个心理阻力大。要提供足够强的不可替代价值才能突破。
Platform Tax
平台税 · Ch.V
依赖别人 API 要付的成本。OpenAI 涨价你毛利就跌，OpenAI 限流你服务就挂。

概念验证 · Ch.V

GPU Poor
算力贫困 · Ch.V

Toxicity
有毒内容 · Ch.VI

Safety Filter
安全过滤器 · Ch.VI

Model Card
模型说明书 · Ch.VI

Transparency Report
透明度报告 · Ch.VI
公开 AI 系统被滥用情况、安全事件、内容审核数据。建立公众信任。
Switching Cost
迁移成本 · Ch.V
用户从你切到竞品的成本：积累的 Prompt / 历史对话 / 工作流集成。
